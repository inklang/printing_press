# Design: Copy Propagation + Algebraic Simplification SSA Passes

**Date:** 2026-03-24
**Status:** Approved

## Context

The Inklang SSA optimization pipeline currently runs three passes in order:

1. Constant Propagation (SCCP-like) — finds SSA values with known constant values, folds constant expressions, eliminates dead branches with constant conditions
2. Global Value Numbering (GVN, intra-block) — detects redundant expressions within a block, replaces duplicates with `Move` instructions pointing at the canonical value
3. Dead Code Elimination (DCE) — removes unreachable blocks and instructions that define unused values

Each pass is run to its own fixed point before the next begins.

Two gaps remain after this pipeline that leave unnecessary instructions in the generated bytecode:

- **Non-constant copies**: GVN produces `Move { dst, src }` when it deduplicates a non-constant expression. CP propagates through `Move` only when `src` is a known constant — it converts `Move { dst, src_constant }` to `LoadImm`. When `src` is a variable, the `Move` survives into the final bytecode. CopyProp handles the remaining non-constant `Move` instructions.
- **Identity/annihilator patterns**: expressions like `x + 0`, `x * 1`, `x && true`, `x || false` survive CP because `x` is not a constant — CP can only fold when both operands are constants. These remain as full `BinaryOp` instructions.

## Solution

Two new SSA passes:

1. `SsaCopyPropagationPass` — eliminates `Move` instructions by replacing all downstream uses of the destination with the source
2. `SsaAlgebraicSimplificationPass` — reduces binary operations where one operand is a known identity or annihilator constant

### Updated Pipeline

```
CP → GVN → AlgebraicSimpl → CopyProp → DCE
```

Rationale for ordering:
- CP runs first to discover as many constants as possible
- GVN deduplicates redundant expressions, producing `Move` instructions; running GVN before AlgebraicSimpl reduces the number of `BinaryOp` instructions AlgebraicSimpl needs to examine
- AlgebraicSimpl uses constants discovered by CP to reduce identity/annihilator patterns, producing `Move` instructions
- CopyProp eliminates all `Move` instructions produced by both GVN and AlgebraicSimpl
- DCE cleans up any instructions that became dead after copy propagation

Each pass still runs to its own fixed point.

## Pass 1: Copy Propagation

**File:** `src/inklang/ssa/passes/copy_propagation.rs`

SSA form makes copy propagation clean: each value is defined exactly once, so if `v2 = Move v1`, every use of `v2` can unconditionally be replaced with `v1` regardless of control flow.

**Algorithm:**

1. Walk all blocks across the function and build a global copy map: for each `Move { dst, src }`, record `dst → src`. Skip any `Move` where `src.is_undefined()` or `defined_value.is_undefined()` — these are SSA sentinel values and must not be propagated.

2. Resolve chains transitively: if `v3 → v2` and `v2 → v1`, collapse to `v3 → v1`. Use a `HashSet` of visited values when following each chain to detect and break cycles (cycles should not arise in well-formed SSA, but must be handled defensively). Only update the map if the chain resolved cleanly — if a cycle is detected, leave `copy_map[dst]` unchanged.

   ```
   for each (dst, src) in copy_map:
       let mut current = src
       let mut visited = HashSet::new()
       visited.insert(dst)
       while copy_map.contains_key(current) && !visited.contains(current):
           visited.insert(current)
           current = copy_map[current]
       if !visited.contains(current):
           // Clean resolution — current is the ultimate source
           copy_map[dst] = current
       // else: cycle detected — leave copy_map[dst] unchanged
   ```

   The `!visited.contains(current)` guard after the loop is critical: the while loop exits either because `current` is not in the copy map (clean resolution) or because `current` is already in `visited` (cycle). Only in the clean case should `copy_map[dst]` be updated — unconditionally writing `copy_map[dst] = current` in the cycle case would set it to an arbitrary mid-cycle node.

3. Walk all blocks again. For every instruction operand (use-site `SsaValue`), if it appears in the copy map, replace it with the mapped value. This includes:
   - All operand fields in `SsaInstr` variants (src, src1, src2, obj, index, etc.)
   - `phi.operands` values (the `SsaValue` on the incoming-edge side of each phi)
   - **Never** rewrite the definition side: `defined_value` fields in instructions and `phi.result` are definitions, not uses. Only the use-side operand values are rewritten.

4. DCE (already in the pipeline) removes the now-dead `Move` instructions.

Cross-block propagation is safe because SSA values are uniquely defined. The copy map is global across all blocks.

**What it produces:** after the subsequent DCE pass, no non-constant `Move` instructions remain (constant `Move`s are already handled by CP converting them to `LoadImm`).

## Pass 2: Algebraic Simplification

**File:** `src/inklang/ssa/passes/algebraic.rs`

Targets binary operations where one operand is a known identity or annihilator constant. These survive CP because the other operand (`x`) is not a constant.

**Algorithm:**

1. Build a `SsaValue → Value` map by scanning `LoadImm` instructions across all blocks. For each `LoadImm { defined_value, const_index }`, look up `constants[const_index]` and record `defined_value → constants[const_index]`. Note: `Value` does not implement `Hash`, so it cannot be used as a map key. The map direction must always be `SsaValue → Value` (keyed on `SsaValue`).

2. For each `BinaryOp { dst, op, src1, src2 }`, check if either operand maps to a relevant constant and apply a simplification rule.

**Identity rules** (always applied, produce `Move { dst, src: x }`):

| Pattern | Condition | Note |
|---|---|---|
| `x + 0` | src2 maps to numeric zero | |
| `0 + x` | src1 maps to numeric zero | |
| `x - 0` | src2 maps to numeric zero | NOT commutative: `0 - x` is `-x`, not `x` |
| `x * 1` | src2 maps to numeric one | |
| `1 * x` | src1 maps to numeric one | |
| `x / 1` | src2 maps to numeric one | NOT commutative: `1 / x` is NOT `x` |
| `x && true` | src2 maps to `Boolean(true)` | |
| `true && x` | src1 maps to `Boolean(true)` | |
| `x \|\| false` | src2 maps to `Boolean(false)` | |
| `false \|\| x` | src1 maps to `Boolean(false)` | |

"Numeric zero" means `Int(0)`, `Float(0.0)`, or `Double(0.0)`. "Numeric one" means `Int(1)`, `Float(1.0)`, or `Double(1.0)`. Subtraction and division are NOT commutative — only the exact forms listed are identities. The reverse forms `0 - x` and `1 / x` are intentionally absent from the table and must not fire.

**Annihilator rules** (produce `LoadImm { dst, const_index }`):

Only applied when the result constant already exists in `ssa_func.constants`. The implementer must search `constants.iter().position(|c| c == &result_value)`. If `None` is returned, skip the rule entirely — no out-of-bounds index may be fabricated.

| Pattern | Condition | Result constant |
|---|---|---|
| `x * 0` or `0 * x` | other src maps to numeric zero | `Int(0)`, `Float(0.0)`, or `Double(0.0)` matching the non-zero operand's type |
| `x \|\| true` or `true \|\| x` | other src maps to `Boolean(true)` | `Boolean(true)` |
| `x && false` or `false && x` | other src maps to `Boolean(false)` | `Boolean(false)` |

**Known limitations:** `x - x → 0` is not handled (would require the result constant `0` to be guaranteed in the table). `x / x → 1` is intentionally excluded — it is only valid when `x ≠ 0`, which cannot be guaranteed without range analysis. These can be addressed as a follow-up with appropriate guards.

The `Move` instructions produced by identity rules feed into CopyProp in the next pipeline stage.

## Implementation Details

### SsaOptPass trait

Both passes implement the existing `SsaOptPass` trait from `src/inklang/ssa/passes/mod.rs`:

```rust
pub trait SsaOptPass {
    fn name(&self) -> &str;
    fn run(&mut self, ssa_func: SsaFunction) -> SsaOptResult;
}
```

### Module registration

Both passes are added to `src/inklang/ssa/passes/mod.rs` as public submodules and wired into `run_optimization_passes` in `src/inklang/ssa/mod.rs` in the order: CP → GVN → AlgebraicSimpl → CopyProp → DCE.

## Testing

Each pass gets unit tests in its own file.

**Copy Propagation:**
- Direct copy: `v2 = Move v1`, downstream use of `v2` is replaced with `v1`
- Chain: `v3 = Move v2`, `v2 = Move v1` → downstream use of `v3` replaced with `v1`
- Cross-block: copy defined in block A, used in block B — confirm the pass is global, not intra-block
- Phi operands are updated: phi incoming value `v2` replaced with `v1` when `v2 = Move v1`
- Phi result `phi.result` is NOT rewritten — only the operand side
- `UNDEFINED` values are not propagated: `Move` with `src.is_undefined()` is skipped
- Non-move instructions are untouched
- `changed = false` when no replacements were actually performed (not just when no moves are present — a move may exist but be dead)

**Algebraic Simplification:**
- Each identity rule fires with `x` as `src1` and as `src2` separately (to test both commuted forms)
- Non-commutative identity forms (`x - 0`, `x / 1`) fire correctly; commuted forms (`0 - x`, `1 / x`) do NOT fire
- Each annihilator rule fires when the result constant is in the table
- Each annihilator rule does NOT fire when the result constant is absent from the table
- Mixed numeric types (e.g., `x: Float + Int(0)`) are handled correctly — numeric zero/one check covers all three numeric types
- `changed = false` when no simplifiable instructions present

Integration tests via the existing `test_ssa_*` tests in `src/inklang/ssa/mod.rs` verify the full pipeline still produces correct output after adding the new passes.
